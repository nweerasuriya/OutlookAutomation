01 — Overview

What changes and what doesn't

Phase 2 is a targeted extension, not a rewrite. The underlying sending logic, CSV management, template engine, and OneDrive sync are entirely unchanged. The work is concentrated in two files — main.py and email_sender.py — and the GitHub Actions workflow.

main.py modified

Replaced single-list env vars with a LISTS_CONFIG JSON array. Now loops over all configured lists in sequence, handling errors per-list without aborting the whole run.

email_sender.py modified

Added an optional list_id parameter to run(). Tags every log line with the list name so multi-list log files are easy to read. Subject lines now also support {{placeholders}}.

email_automation.yml modified

Swapped individual email/CSV/template secrets for LISTS_CONFIG. Timeout increased to 12 hours to accommodate multiple lists. Added a list_filter manual input for targeted testing.

csv_manager.py unchanged

No changes required. Each list has its own CSV file and its own CSVManager instance — isolation is automatic.

template_engine.py unchanged

No changes required. The template resolution logic added in the Phase 1 amendment already handles any filename correctly.

onedrive_sync.py unchanged

No changes required. sync_down already downloads every file in the csv/ and templates/ folders — new lists just mean new files land in the same places.

02 — Architecture

Architecture shift

In Phase 1, each env var drove a single list. In Phase 2, a single LISTS_CONFIG JSON secret contains an array of list definitions. One GitHub Actions run processes all of them sequentially — one token, one OneDrive sync, then a loop.

Why sequential, not parallel?

Parallel jobs would finish faster but create race conditions on the CSV files during the upload phase — two jobs writing the same OneDrive file simultaneously causes data loss. Sequential processing is slower but safe. The 5-minute interval between sends per list already makes the bottleneck the send rate, not the processing.

One Azure app, many senders

The Mail.Send application permission granted in Phase 1 covers all mailboxes in the tenant. You do not need to create a new Azure app per sender. Every list in LISTS_CONFIG can specify a different sender_email and the same token will work for all of them — provided they are all in the same Microsoft 365 tenant.

⚠ If a future client is on a different Microsoft 365 tenant (i.e. a completely separate company), they will need their own Azure app registration, their own set of Azure secrets, and ideally a separate GitHub repository or workflow file. Phase 2 assumes all senders are within a single tenant.
03 — File-by-file changes

Exactly what changed in each file

main.py — the core change

The single-list environment variables (SENDER_EMAIL, CSV_FILENAME, etc.) are gone. They are replaced by a load_lists_config() function that parses a JSON array from the LISTS_CONFIG secret, then main() loops over each entry.

main.py — key structural change
remote_base = os.environ["ONEDRIVE_REMOTE_BASE"] csv_filename = os.environ["CSV_FILENAME"] template_filename = os.environ["TEMPLATE_FILENAME"] sender_email = os.environ["SENDER_EMAIL"] email_subject = os.environ["EMAIL_SUBJECT"]   configs = load_lists_config() # parses LISTS_CONFIG JSON secret for cfg in configs: # loop over every list send_emails( # one call per list csv_path=..., sender_address=cfg["sender_email"], list_id=cfg["list_id"], )

A key design decision: a failure in one list is caught and logged but does not abort the remaining lists. Every list gets its chance to run regardless of what happened before it.

04 — LISTS_CONFIG

The LISTS_CONFIG secret

This is the single most important change to understand. The entire multi-list configuration lives in one GitHub secret as a JSON array. Each object in the array defines one list.

LISTS_CONFIG value (stored as GitHub secret)[
  {
    "list_id":           "industry_a",       // unique identifier, used in logs
    "sender_email":      "alice@company.com", // Outlook address to send from
    "email_subject":     "A note for {{company_name}}",
    "csv_filename":      "industry_a.csv",   // must exist in OneDrive csv/ folder
    "template_filename": "industry_a.html",  // must exist in OneDrive templates/ folder
    "daily_limit":       25,                 // optional, default 25
    "interval_seconds":  300                 // optional, default 300 (5 min)
  },
  {
    "list_id":           "industry_b",
    "sender_email":      "bob@company.com",
    "email_subject":     "Reaching out",
    "csv_filename":      "industry_b.csv",
    "template_filename": "industry_b.html"
  }
]
💡 Write and validate your JSON at jsonlint.com before pasting it into GitHub. A single misplaced comma will cause the entire run to fail with a clear parse error in the logs.

Adding a new list

To add a third or fourth list later: edit the LISTS_CONFIG secret in GitHub, append a new JSON object to the array, upload the new CSV and template to OneDrive, and the next scheduled run will pick it up automatically. No code changes needed.

Removing a list

Remove its entry from the JSON array and the system ignores it from the next run. You can keep the CSV and template files in OneDrive for archival purposes — they'll just never be touched.

05 — OneDrive

OneDrive folder structure

The folder structure is unchanged — you're just adding more files to the same folders. onedrive_sync.py downloads every .csv in csv/ and every .html in templates/, so new lists are automatically included.

email-automation/ ← ONEDRIVE_REMOTE_BASE (unchanged) ├── csv/ │ ├── industry_a.csv ← Phase 1 list (renamed from contacts_list1.csv) │ ├── industry_b.csv ← new list — just upload to this folder │ ├── industry_c.csv ← another new list │ └── backups/ ← monthly backups, auto-created ├── templates/ │ ├── industry_a.html ← existing template │ ├── industry_b.html ← new template for list B │ └── industry_c.html ← new template for list C └── logs/ ← unchanged
ℹ The csv_filename and template_filename values in LISTS_CONFIG must match the filenames in OneDrive exactly, including case. OneDrive paths are case-sensitive via the Graph API.

07 — Migration

Step-by-step migration

1
Replace the three changed files

Copy the Phase 2 versions of src/main.py, src/email_sender.py, and .github/workflows/email_automation.yml over the Phase 1 versions in your repo. The other four source files are untouched.

2
Add the test file

Copy tests/test_phase2.py into your tests/ folder alongside the existing Phase 1 tests. Both test files will run together on every workflow trigger.

3
Rename your existing CSV and template (optional but recommended)

If your Phase 1 CSV was called contacts_list1.csv, rename it to something descriptive like industry_a.csv — both in OneDrive and in your LISTS_CONFIG value. This is optional but makes the logs much easier to read as you add more lists.

4
Build the LISTS_CONFIG JSON

Write the JSON array for all your current lists — starting with just the one from Phase 1, then add the new ones. Validate it at jsonlint.com. Then go to your GitHub repo → Settings → Secrets → Actions → New secret. Name it LISTS_CONFIG and paste the JSON.

5
Upload new CSVs and templates to OneDrive

For each new list, upload its .csv to the csv/ folder and its .html to the templates/ folder in OneDrive. File names must match exactly what's in LISTS_CONFIG.

6
Do a dry run

Push the changes, then go to Actions → Email Outreach Automation → Run workflow → set dry_run = true. This runs the full test suite including the Phase 2 tests, then exits without sending anything. Check the logs confirm all lists are detected and the config parses correctly.

7
Live test with one list only (optional)

For extra caution, temporarily set LISTS_CONFIG to just the one Phase 1 list you know works, do a live run, confirm it sends and updates the CSV. Then expand back to all lists.

8
Remove old secrets

Once you've confirmed a full multi-list run works cleanly, delete SENDER_EMAIL, EMAIL_SUBJECT, CSV_FILENAME, and TEMPLATE_FILENAME from GitHub Secrets. They're no longer referenced anywhere.

08 — Testing

Testing the transition

The Phase 2 test file (tests/test_phase2.py) covers:

Config validation

Tests that load_lists_config() correctly rejects missing env vars, invalid JSON, empty arrays, and configs missing any required key. Every required key is tested individually via @pytest.mark.parametrize.

Template resolution

Confirms the OneDrive-synced copy takes priority over the repo sample, the fallback works when OneDrive copy is missing, and a clear error is raised when neither exists.

Subject placeholder substitution

Verifies that {{company_name}} and {{first_name}} in subject lines are filled correctly per contact.

CSV isolation

Confirms that two CSVs load independently and that saving one does not affect the other — important for catching any accidental shared state.

Run locally at any time with:

pytest tests/ -v --tb=short
09 — Scheduling

Timeout & scheduling maths

With multiple lists, the GitHub Actions job runs much longer. Use this formula to set timeout-minutes appropriately:

# Time per list (minutes)
time_per_list = (daily_limit × interval_seconds) / 60

# Total job time
total = sum(time_per_list for each list) + 30  # +30 min buffer

# Example: 3 lists × 25 emails × 300s each
(25 × 300) / 60 = 125 min per list
125 × 3 + 30    = 405 min total  → set timeout-minutes: 420
⚠ GitHub Actions has a hard limit of 6 hours (360 minutes) for free accounts and 35 days for paid — but in practice jobs over 6 hours are unusual. If your total send time genuinely exceeds 6 hours, consider splitting into two scheduled workflow files that run at different times of day.

Staggering lists across the day

An alternative to running all lists in one long job is to create separate workflow files with different cron times — e.g. list A at 9 AM, list B at 11 AM. This keeps individual jobs short, makes logs easier to read, and means one list failing doesn't delay the others. The trade-off is more workflow files to maintain. For 2–3 lists the single-job approach is simpler; beyond that, split workflows are worth considering.
